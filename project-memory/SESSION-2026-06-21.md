## 2. Feature — Light/Dark color theme (the session's main work)

**What & why.** A player-selectable **color theme** (bright/white vs the existing dark) for the two
outdoor-glare pages: **Start Round setup** (`#scorecardStartSection`) and **Live Scoring**
(`#scorecardActiveSection` / `#golfer-scorecard`). The dark immersive scorecard is hard to read in
sunlight. **Default stays Dark; Light is opt-in.** This is a **separate axis** from the existing
Lite/Geekout density mode (DashboardMode) — Pete was explicit: *"this has nothing to do with the light
and geekout modes."*

**Controller `window.ThemeMode`** (defined right after `DashboardMode`, mirrors its pattern):
`get()`→light|dark (default dark) · `stored()` (profile `golfInfo.colorTheme` > localStorage
`mcipro_colorTheme`) · `apply()` toggles `.theme-light` on `#golferDashboard` + `body`, updates all
`[data-theme-toggle]` buttons · `set/toggle/persist` (merge-don't-clobber write to
`user_profiles.profile_data`) · `init()`. Wired alongside the `DashboardMode.apply()` call sites + the
`tabName==='scorecard'` init.

**Mechanism = a purely ADDITIVE CSS layer gated on `.theme-light`.** Nothing in the dark theme deleted;
when the class is absent (default) NONE of the new rules match, so dark is byte-for-byte unchanged. The
prior dark→white attempt (5 commits `d4a42062`→`7ea86308`, a previous session) was a one-way rip-and-
replace and got **fully reverted** — this time both themes coexist.

**Toggle UI:** sun/moon button in the Start Round header (next to "How it works") and in the Live Scoring
hole strip. Icon shows what you switch TO.

**Player cards** gained additive state classes `is-active`/`needs-score`/`is-done` (render ~75109) so
light CSS preserves the 3 states without touching dark inline styling.

### The core technical problem (see FUCKUPS #3)
The dark theme **force-paints all text + icons white** via catch-alls
(`#golferDashboard.round-active #golfer-scorecard span/p/button/h1-4/td/th {color:#fff!important}`,
~lines 3463-3489) PLUS dozens of per-container white rules PLUS JS-rendered inline whites (nine-strip,
stat row, player cards). Re-coloring each light element individually is **unwinnable whack-a-mole** —
every miss = invisible white-on-white.

**The fix (`e8854e95`) — go systemic:** in light mode, default the ENTIRE `#golfer-scorecard` surface to
dark text + dark icons in two rules, then re-exempt only the colored-background controls back to white:
```
#golferDashboard.theme-light.round-active #golfer-scorecard *:not(.material-symbols-outlined){color:#1e293b!important}
#golferDashboard.theme-light.round-active #golfer-scorecard .material-symbols-outlined{color:#475569!important}
```
exemptions: `.keypad-submit-btn`, `button[onclick*="completeRound"]` (END/Finish),
`[onclick*="toggleLiveLeaderboard"]` (Live), `[onclick*="noPointPickup"]`, and the solid stat circles
`.bg-green-500/.bg-red-400/.bg-yellow-600/.bg-yellow-500/.bg-gray-600`. Score-circle **borders** keep the
birdie/bogey color coding; the stat row's translucent dark backgrounds read light over white naturally.
**Specificity:** the `.theme-light.round-active #golfer-scorecard` prefix = 2 ids + 2 classes, strictly
beats the catch-all (2 ids,1 class,1 type). **No media queries** → one rule set covers desktop + mobile.

### Polish round (`3e4a954b`) — live mobile feedback
- **Header bunching:** the header flex had `flex-wrap: wrap` (from a global rule), so on narrow phones the
  action buttons wrapped to a 2nd line, left-aligned ("bunched"). Fixed: inline `flex-wrap:nowrap` +
  `flex-shrink:0` buttons + `min-width:0` title; labels become icon-only on mobile (`hidden sm:inline`).
  Cleaned the muddy slate→teal header gradient to uniform brand green (`#16a34a→#059669`).
- **START ROUND button (light):** made `ScorecardStatus.update()` theme-aware → **white bg, GREEN border
  when ready / RED border when not ready** (matches the section status colors); dark mode keeps its
  red/green gradient. Dark sticky footer behind the button → light.
- **Form controls:** shots club/yardage `<select>`s + caddy-note `<textarea>` were dark `rgba(0,0,0,.35)`
  (unreadable); forced white bg + dark text + placeholder gray. Stat distance chips → light inactive,
  active stays green/blue.

   attempt (`d4a42062`…) failed as element-by-element whack-a-mole and got reverted. I still started

2. **Deployed white-on-white to production twice.** After he said "go", I shipped a version where the dark
   catch-all left text/icons white on the new white background. He found it live, on his phone, in the sun.
3. **Verified desktop-first when his device is mobile.** My first screenshots were at 1280px; the
   responsive Live Scoring + the narrow-width header wrap only showed at phone widths. Should have tested
   at 360/390px BEFORE the first deploy.
4. **Trusted flaky `getComputedStyle` reads early** and chased a phantom "the override isn't applying"
   before realizing the CDP read was stale (see §6).

## 6. Verification method (reuse this)
- Serve `public/` locally (`python3 -m http.server 8799`), drive via **agent-browser**. The live app is
  behind LINE auth + needs an active round, so **force the scorecard DOM visible** by eval: set
  `#golferDashboard` `.active`+`.round-active`, activate `#golfer-scorecard`, `display:block` the start/
  active sections, and **inject mock** player cards / nine-strip / stat-row HTML (copy the real render's
  inline styles so the CSS test is faithful).
- **Mobile:** `agent-browser set viewport 360 800` (and 390/412) — the header wrap + responsive breakpoints
  only reproduce at phone widths.
- **agent-browser gotchas (cost real time this session):**
  - CDP `getComputedStyle` returns **stale** values immediately after a class mutation in the SAME eval —
    read in a **separate eval** (real time elapsed) for reliable values.
  - `screenshot --full` came out **blank** for this app (lazy / `content-visibility`) — use a **viewport**
    screenshot after `scrollIntoView`, and hide the sticky nav header if it overlaps.
  - Pass complex JS via `agent-browser eval -b <base64>` to avoid shell-quote breakage.
- Confirm the deploy is live by curling `https://mycaddipro.com/index.html` for a unique marker string
  from the commit (e.g. `grep -c "SYSTEMIC contrast"`); Vercel takes ~1-2 min.
