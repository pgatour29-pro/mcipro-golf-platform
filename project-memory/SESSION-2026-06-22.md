## 2. The problem

Pete (as a player) messaged the JOA organizer through an event card's private thread, and the
organizer never saw it. Root cause was **not** a data bug — the message wrote fine. The private-thread
feature (built 2026-06-19) was **pull-only**: the organizer had to open the per-event 📢 modal to find
"Player Questions". There was no badge on any main surface and **no push** of any kind. So unless the
organizer drilled two taps into the right buried modal, they never knew a message existed.

## 3. What shipped — message badges (3 surfaces)

The unread count for event notices + private messages now appears on every "main" events surface:

1. **Player** — golfer dashboard **Society Events cube** (`.enb-events-badge`, all 3 layouts: lite /
   full / caddie). Counts unread organizer *notices* (on registered events) + unread organizer
   *replies* in the player's own threads. For a user who is also an organizer, it folds in org-side
   unread too (via `_myOrgEventIds()`, which resolves the user's organized events independently of the
   org dashboard being open).
2. **Organizer** — the **Events tab** (`.enb-orgtab-badge`, mobile + desktop) = total unread player
   questions across the organizer's events.
3. **Organizer** — the **org-home Events cube** (the Society Organizer LITE home cube grid). Same
   `.enb-orgtab-badge`, hydrated via a hook in `updateOrgLiteCubes`.

All hydrate on load + the existing 15s `EventNoticeBoard` poll. **Self-contained — deliberately does
NOT route to the Messages module inbox** (Pete's explicit requirement). One-way per message: the
**sender never badges their own message**; only the recipient is notified.

## 4. What shipped — LINE push on each message

`EventNoticeBoard._notifyThread` fires from `sendThread` after a successful insert and pushes the
counterparty via the existing `system_alert` path (`line-push-notification` edge fn):

- **Player asks** → organizer (recipient resolved from `society_profiles.notify_line_id || organizer_id`).
- **Organizer replies** → the player who owns the thread.
- Routes `KAKAO-*` recipients to the new `kakao-push` fn, `U*` to LINE. No-ops gracefully for ids with
  no push channel.

## 5. The JOA / Jason identity problem

Jason (JOA organizer) logs in via **Kakao** (`KAKAO-4911042963`), so `society_profiles.organizer_id`
must stay his Kakao id (it's the login/dashboard gate) — a LINE push can't target it. But he **also has
an older LINE account** (`Udb12b92…`, "Jason Kang") that is **push-capable** (it has a
`messaging_user_id`, i.e. he'd added the MyCaddiPro LINE bot). Fix: added a dedicated
**`society_profiles.notify_line_id`** column = "where to send organizer notifications", decoupled from
the login id, and set JOA's to Jason's LINE account. `_notifyThread` prefers it. So JOA questions reach
Jason on LINE **now**, without waiting on Kakao activation. General mechanism for any
Kakao/Google-login organizer with a usable LINE account.

## 6. Kakao push (built, PARKED)

For organizers with **no** LINE account, built the Kakao path via the **"send to me" (memo) API**:
- `kakao_push_tokens` table (RLS-locked, service_role only) stores each Kakao user's OAuth token.
- `kakao-oauth-exchange` now persists the token on login; login requests the `talk_message` scope.
- New `kakao-push` edge fn: looks up the token, refreshes if expired, sends the memo.

**Not active until Pete does two manual things** in the Kakao Developers console (app `b11fe825…`):
(1) enable the `talk_message` consent item + activate the "send to me" Message product; (2) Jason
re-logs in once to grant it. If the app is still in dev mode, Kakao restricts `talk_message` to
registered team members. **Parked at Pete's request** — JOA uses the LINE path above in the meantime.
